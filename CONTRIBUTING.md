## Contributing to the Project

Thank you for your interest in contributing to the Typst Docs Project! It contains two main components:

1. [`typst-docs-l10n`](./crates/typst-docs-l10n/): This is the localization tool that contains the translations for the Typst documentation.
2. [`locales/docs`](./locales/docs/): This directory contains the localized documentation files in various languages.

### Contributing to `typst-docs-l10n`

Please install the Rust toolchain. Then you can build the `typst-docs-l10n` crate using the following command:

```bash
cargo build crates/typst-docs-l10n
```

See the [README.md](./README.md) for usage of the `typst-docs-l10n` CLI.

### Contributing to `locales/docs`

To contribute to the localized documentation files, you only need to edit the files in the `locales/docs` directory, whose keys are generated by `typst-docs-l10n`, and values are _Markdown_ documentation extracted from [typst](https://github.com/typst/typst). The files are generated suitably for LLM-based translations.

Use [translation-principle.md](./docs/translation-principle.md) as the shared
translation policy and [terms.toml](./docs/terms.toml) as the canonical
glossary for recurring or ambiguous terminology. For machine-specific guidance,
see [copilot-instructions.md](./.github/copilot-instructions.md).

To follow the console-first workflow below, install the Rust toolchain for the
scan helper, Node.js with `npm` for validation, and GitHub CLI only if you want
to open the pull request from the terminal.

#### Console-First Translation Iteration

One translation iteration should stay review-sized. Prefer a single entry, a
single included body file, or a small cluster of related keys over a broad
backlog sweep.

1. Scan for candidate work with the local helper:

   ```bash
   cargo run -p translate -- scan
   ```

   The helper reports where missing or stale translation work lives. Re-run
   with `--limit 0` for the full backlog or `--format json` when structured
   output is easier to inspect.

2. Pick one review-sized target and locate its storage:
   - If the helper points at a top-level key such as `key=index.title`, edit
     [`locales/docs/typst-docs.toml`](./locales/docs/typst-docs.toml).
   - If the matching `en` value in that file is `{{typst-docs/<file>.toml}}`,
     edit the included body file under
     [`locales/docs/typst-docs/`](./locales/docs/typst-docs/).
   - If the helper reports `paragraph=main.<index>`, edit only that `[[main]]`
     entry in the included body file unless you are intentionally syncing a
     wider source change.

3. Edit minimally:
   - Preserve TOML structure, Markdown, Typst code, links, labels,
     placeholders, and unrelated surrounding content.
   - Reuse entries from [terms.toml](./docs/terms.toml) when a recurring term
     already has a canonical translation.
   - Add or refine a glossary entry when a recurring term does not yet have a
     stable translation or needs clearer context.

4. Validate and review the patch:

   ```bash
   npm run validate
   git diff -- <files>
   ```

   Inspect the `npm run validate` output manually. The current validator logs
   TOML parse errors but does not fail with a non-zero exit code yet, so the
   command is not a hard gate by itself.

5. Open a pull request from the console when your local GitHub CLI setup is
   ready:

   ```bash
   gh pr create
   ```

   This step requires `gh` to be installed, authenticated, and authorized for
   this repository.
